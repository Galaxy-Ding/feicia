# 需求报告

## 1. 项目名称

废才 Seedance 2.0 AI 分镜师团队复刻实施项目

## 2. 项目背景

当前项目已经定义了一套高密度 Prompt 工程方法，用于把剧本逐步加工为：

- 导演讲戏本
- 人物设定提示词
- 场景环境提示词
- Seedance 2.0 动态视频提示词

其核心价值不在于“单次生成”，而在于：

- 有明确分工的多 Agent 协作
- 有状态管理的分阶段编排
- 有业务与合规双审核闭环
- 有跨集复用的素材资产沉淀

目标是把这套方法工程化为一个可运行的本地项目，使用户接入自己的大模型 API 后，即可按同样方式生产分镜资产。

## 3. 业务目标

1. 将剧本输入标准化为一套稳定可复制的 AI 分镜生产流程。
2. 确保每集产物文件结构固定、命名固定、输出标准固定。
3. 降低对人工导演、服化道、分镜师协作成本的依赖。
4. 在不丢失原始 Prompt 方法论的前提下，实现可反复执行、可审计、可追踪。

## 4. 目标用户

- 拥有 OpenAI 兼容或其他 LLM API 的个人创作者
- 小型 AI 短剧/短片团队
- 需要批量生产剧本分镜提示词的内容工作室

## 5. 核心使用场景

### 场景 A：首集从零开始

用户放入 `script/ep01-xxx.md`，系统识别当前集数，执行导演分析，审核通过后再进行服化道设计，再进入 Seedance 分镜阶段。

### 场景 B：跨集生产

用户继续放入 `ep02-xxx.md`，系统对比既有 `assets/` 中的人物与场景，自动标注“新增 / 复用 / 变体”。

### 场景 C：中途修改

用户在任意阶段提出修改意见，系统应判断影响范围，恢复对应 Agent 上下文，修改文件并重新审核。

### 场景 D：项目状态追踪

用户不记得做到哪一集哪一阶段时，系统可通过扫描 `script/`、`outputs/`、`assets/`、`.agent-state.json` 恢复进度。

## 6. 功能需求

### FR-01 剧本输入管理

- 系统应支持从 `script/` 读取 `.md` 或 `.txt` 剧本文件。
- 文件名应支持识别 `ep01`、`ep02` 等集数标识。
- 系统应支持单集和多集项目。

### FR-02 项目状态检测

- 系统应自动扫描 `script/` 与 `outputs/` 判断每集所处阶段。
- 系统应识别当前默认处理集数。
- 系统应能输出项目进度摘要。

### FR-03 三阶段执行

- 阶段一：导演分析，生成 `outputs/<集数>/01-director-analysis.md`
- 阶段二：服化道设计，追加写入 `assets/character-prompts.md` 与 `assets/scene-prompts.md`
- 阶段三：分镜编写，生成 `outputs/<集数>/02-seedance-prompts.md`

### FR-04 双审核闭环

- 每个阶段完成后，必须执行业务审核。
- 业务审核通过后，必须执行合规审核。
- 任一审核失败，系统必须合并修改意见，回到对应执行 Agent 进行一次性修订。

### FR-05 Agent 上下文恢复

- 同一集内重复调用同一子 Agent 时，应恢复此前对话上下文。
- 跨集时应重置子 Agent 状态，避免上下文污染。
- 状态应保存在 `.agent-state.json`。

### FR-06 素材沉淀与跨集复用

- 系统应在导演阶段识别人物和场景素材状态。
- 服化道阶段仅为“新增/变体”项生成提示词。
- 分镜阶段应按现有素材文件建立 `@图片` 素材映射表。

### FR-07 修改请求处理

- 用户可在任意阶段追加修改意见。
- 系统应识别受影响文件和回流阶段。
- 修改后应重新触发该阶段双审核。

### FR-08 命令路由

- 系统应支持 `~start`
- 系统应支持 `~design`
- 系统应支持 `~prompt`
- 系统应支持 `~status`
- 系统应支持 `~help`

### FR-09 日志与可追踪性

- 系统应记录关键操作日志。
- 系统应能定位每个文件由哪个阶段生成、何时修改、为何修改。

### FR-10 模型与 API 可配置

- 系统应允许用户配置不同角色的模型名、Base URL、API Key、温度、最大 Token 等参数。
- 系统应允许单模型部署和多模型部署两种模式。

## 7. 非功能需求

### NFR-01 一致性

- 相同输入和相近模型配置下，应获得结构稳定的输出。

### NFR-02 可维护性

- Prompt 文件与执行引擎分离。
- 角色和技能以文件形式管理，便于升级。

### NFR-03 可扩展性

- 后续可增加更多审核 Skill、更多导出格式、更多命令。

### NFR-04 可审计性

- 每次审核失败的原因必须可追溯。

### NFR-05 低耦合

- 不将业务规则硬编码到主程序中，优先由 Markdown Prompt 文件驱动。

## 8. 约束条件

- 需要用户自备大模型 API。
- Seedance 平台存在素材数量上限和内容限制。
- Gemini/Nano Banana 对图像提示词存在较高的合规误杀风险。
- 原项目是中文工作流，所有产物必须使用中文。

## 9. 成功标准

满足以下条件即可判定需求达成：

1. 单集全流程跑通并产出 4 类标准文件。
2. 至少一个跨集案例能正确复用素材。
3. 至少一个审核失败案例能完成自动回修并通过复审。
4. 至少一个用户修改请求能正确回流到受影响阶段。
