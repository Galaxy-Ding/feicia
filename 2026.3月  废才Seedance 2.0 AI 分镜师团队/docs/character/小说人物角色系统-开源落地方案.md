## 1. 文档目的

本文件回答三个问题：

- 针对“小说人物角色系统”，哪些开源路线是真正成熟、稳重、可落地的
- 你列出的 `LangGraph / AutoGen / CrewAI / Haystack / Flowise / LlamaIndex / Dify / OpenClaw` 应该分别放在什么位置
- 如何把你定义的 7 个 Agent 和 10 个 skill 组织成一套可以上线跑书、可人工校对、可追溯证据的工程方案

这里的“可落地”不是指做一个会聊天的 demo，而是：

- 能处理长文本
- 能保留中间状态
- 能中断、恢复、人工介入
- 能输出结构化 JSON 和稳定文档
- 能做证据回链和一致性校验

## 2. 先给结论

### 2.1 最推荐的主干方案

中文小说：

- `LangGraph + HanLP + 规则/LLM 抽取器 + JSON Schema + SQLite/Postgres`

英文小说：

- `LangGraph + BookNLP + 规则/LLM 抽取器 + JSON Schema + SQLite/Postgres`

这两条路线最稳，因为它们把问题拆成两层：

- 上层用 `LangGraph` 做长流程、状态、恢复点、人工审核点
- 下层用 `HanLP` 或 `BookNLP` 做尽量确定性的文本结构化预处理

### 2.2 `OpenClaw` 的正确位置

`OpenClaw` 可以接入这个系统，但不应该做总编排器。

更合理的定位是：

- `OpenClaw` 负责浏览器执行、网页登录态复用、技能封装、外部系统接入
- 人物角色系统的核心抽取与状态流转，仍然由 `LangGraph` 或保守版 `Haystack Pipeline` 承担

换句话说：

- `OpenClaw` 适合做“执行层”
- 不适合做“人物抽取主脑”

### 2.3 备选排序

如果按“适合做人物角色系统主干”的顺序排：

1. `LangGraph`
2. `Haystack`
3. `LlamaIndex`
4. `AutoGen`
5. `CrewAI`
6. `Flowise`
7. `Dify`

补充说明：

- `AutoGen` 和 `CrewAI` 很适合做多角色协作原型，但不如 `LangGraph` 稳
- `Haystack` 很适合做透明、可控、可测的抽取管线
- `LlamaIndex` 更偏文档 agent / 数据 agent，可做辅助手或替代部分抽取层
- `Flowise` 适合先搭可视化 demo，不适合做最终主干
- `Dify` 适合快速产品化试验，但许可证不是最干净的 permissive 路线

## 3. 为什么 `LangGraph` 仍然是主干首选

根据官方文档与官方仓库，`LangGraph` 当前明确强调：

- long-running, stateful agents
- durable execution
- human-in-the-loop
- low-level orchestration

这些能力和你的人物系统是直接对齐的：

- 一本小说往往不是一次调用能做完的
- alias merge、指代消解、导演对齐、中间人工修订都需要暂停和恢复
- QA 阶段必须能回放前序状态，而不是重新整本书乱跑一遍

对你来说，`LangGraph` 最有价值的不是“多 agent 很酷”，而是：

- 有状态图
- 有 checkpoint
- 有 interrupt
- 有线程级恢复能力
- 可以把每个 Agent 输出严格收束为 schema

这正是人物角色系统最难的地方。

## 4. 其他开源路线应该怎么放

### 4.1 `Haystack`

官方定位已经很清楚：

- 它是面向 production-ready LLM applications 的 open-source orchestration framework
- 强项是 modular pipelines、agent workflows、explicit control

它对这个项目的价值是：

- 更像“透明流水线”，不是“聊天型 agent”
- 很适合把 `人物发现 -> 别名归并 -> 台词归属 -> QA` 做成一串可测组件
- 如果团队偏后端工程、偏测试、偏显式组件依赖，`Haystack` 是最稳的备选主干

它不如 `LangGraph` 的地方在于：

- 长流程恢复和人工中断体验不如 `LangGraph` 直观
- 社区里“长文本多 agent 编排”的范式心智不如 `LangGraph` 强

结论：

- 如果你要最稳的“图式状态编排”，选 `LangGraph`
- 如果你要最稳的“组件化抽取流水线”，`Haystack` 是第二名

### 4.2 `LlamaIndex`

官方文档已经把多 agent 路线拆成三种：

- `AgentWorkflow`
- `Orchestrator agent`
- `Custom planner`

它适合这个项目的点在于：

- 文档 ingestion、节点化、结构化抽取、检索增强都比较顺手
- 对“按章节索引、按角色检索证据、按节点维护状态”很友好
- 如果未来要把人物卡、关系卡、证据片段做成可查询知识层，`LlamaIndex` 很方便

它不适合直接做你第一版主干的原因：

- 人物系统的核心难点不是“更强检索”，而是“更稳编排和审校”
- 如果主干已经有 `LangGraph`，再上 `LlamaIndex` 更适合作为文档层和检索层，而不是重复造一个总控

结论：

- `LlamaIndex` 适合做辅助手、索引层、结构化抽取层
- 不建议第一版就把它当总编排器

### 4.3 `AutoGen`

官方仓库当前明确提示：

- 新用户优先看 `Microsoft Agent Framework`
- `AutoGen` 仍会持续维护，并接收 bug fix 和关键安全补丁

这说明它仍然可用，但主力方向已经不再是“所有新系统都首推 AutoGen”。

它适合你项目的部分：

- 快速试导演 Agent、审稿 Agent、角色设定师 Agent 的协作逻辑
- 快速试“多角色会议室”形式的原型

它不适合做人物系统主干的原因：

- 你的核心问题不是“角色对话”
- 而是“强状态、强证据、强恢复、强 QA”

结论：

- `AutoGen` 可以用来验证多角色交互范式
- 不建议做正式生产主干

### 4.4 `CrewAI`

官方仓库和文档的优点是：

- 角色表达清晰
- `Flows` 有 state、event-driven、branching
- 文档里已经补了 telemetry 说明和关闭方式

它适合你的点：

- 很适合把业务语义写得很清楚
- 非技术同学容易看懂“角色分工”

但它的问题也很明显：

- 人物角色系统真正复杂的是 alias merge、证据不足判定、一致性校验
- 这些不是靠“多角色发言”本身解决的

结论：

- `CrewAI` 适合做角色表达层、原型层、业务可读层
- 不建议单独承担主干

附加注意：

- 默认匿名遥测应明确关闭
- 至少设置 `CREWAI_DISABLE_TELEMETRY=true`

### 4.5 `Flowise`

官方文档里，`Agentflow V2` 已经明确转向：

- explicit workflow orchestration
- Flow State
- checkpoint
- human-in-the-loop
- multi-agent communication

这比早期“只是拖节点聊天”成熟不少。

但它仍更适合：

- 给团队快速演示流程
- 做运营/校对台的可视化入口
- 做原型验证

不建议它做主干的原因：

- 真正难的部分仍然在代码层
- 你最终还是要把 alias merge、证据回链、QA 规则写成可测试逻辑

结论：

- `Flowise` 适合 demo、可视化、低代码原型
- 不适合作为第一生产主干

### 4.6 `Dify`

`Dify` 官方仓库当前说明：

- 它是 production-ready platform for agentic workflow development
- 但仓库许可证是 `Dify Open Source License`，基于 Apache 2.0 且附加额外条件

这意味着：

- 它并不是最干净的 permissive 路线
- 如果你在意长期自主可控和许可证清晰度，不应把它放在第一优先级

结论：

- `Dify` 可以做业务原型和内部工具
- 不建议作为你这个系统的首选基础设施

### 4.7 `OpenClaw`

`OpenClaw` 当前最有价值的是：

- `AgentSkills` 兼容 skill 目录
- workspace / shared skill 机制
- 浏览器执行
- 专用 browser profile
- 手动登录后复用登录态
- multi-agent 场景下 per-agent workspace

它对人物系统的正确贡献不是“替代 LangGraph”，而是：

- 跑网页型导演资料采集
- 跑内部 CMS / Notion / 飞书页面资料导出
- 跑审核台截图、回填、发布
- 封装项目私有技能

结论：

- `OpenClaw` 适合作为 browser + skill + external systems layer
- 不适合作为人物抽取总编排器

## 5. 成熟稳重的推荐架构

### 5.1 总体架构

```text
Novel Input
  -> Chapter / Scene Splitter
  -> Evidence Span Indexer
  -> Character Discovery Agent
  -> Coreference / Alias Merge Agent
  -> Dialogue Attribution Agent
  -> Profile Extraction Agent
  -> Director Alignment Agent
  -> Consistency QA Agent
  -> Document Writer Agent
  -> Final Character Bible

Shared Stores
  - chapter_spans
  - character_registry
  - alias_candidates
  - dialogue_registry
  - profile_facts
  - relation_edges
  - evidence_links
  - review_queue
```

### 5.2 推荐的“稳妥技术栈”

编排层：

- `LangGraph`

基础 NLP 层：

- 中文：`HanLP`
- 英文：`BookNLP`

结构化输出层：

- `Pydantic`
- `JSON Schema`

持久化层：

- `SQLite` 用于单机试验
- `Postgres` 用于多人协作和线上版本

检索与证据层：

- 第一版直接用 `Postgres + text span offsets`
- 不建议一开始就上图数据库
- 向量库可以后加，不是第一阶段必须

文档输出层：

- `Jinja2` 或等价模板
- Markdown 为主，必要时导出 HTML / PDF

人工审核层：

- `LangGraph interrupt`
- 简单 Web 审核页或 CLI 审核命令

浏览器与外部系统层：

- `OpenClaw`

### 5.3 为什么这套组合最稳

因为它满足了四个原则：

1. 编排和抽取解耦  
   `LangGraph` 不负责识别人名，`HanLP/BookNLP/LLM` 不负责总状态流转。

2. 先用确定性，再用 LLM  
   能靠 NER、coref、quote attribution、规则完成的，就不要一开始全交给大模型。

3. 所有结论都能回链到证据  
   每一条人物属性、关系、台词归属都要能回到章节、段落、字符区间。

4. 保留人工断点  
   alias merge、关系冲突、导演诠释分歧必须允许人工插手。

## 6. 7 个 Agent 的推荐拆法

| Agent | 主要职责 | 依赖 skill | 主要输出 |
| --- | --- | --- | --- |
| Character Discovery Agent | 找出人名、别名、称谓、首次出场 | 人名识别、章节/场景出场统计、证据回链 | `character_mentions.json` |
| Coreference / Alias Merge Agent | 合并同一角色的名字、别称、代词 | 别名归并、指代消解、证据回链 | `character_entities.json` |
| Dialogue Attribution Agent | 台词归属、说话风格、口头禅、情绪倾向 | 台词归属、章节统计、证据回链 | `dialogue_attribution.json` |
| Profile Extraction Agent | 外貌、年龄段、身份、性格、动机、能力、弱点、关系、关键事件 | 人物属性抽取、人际关系抽取、证据回链 | `character_profiles.json` |
| Director Alignment Agent | 把导演分析稿和客观文本做双层对齐 | 文档读取、证据回链、JSON Schema 输出 | `director_aligned_profiles.json` |
| Consistency QA Agent | 查冲突、查错并、查证据不足 | 一致性校验、证据回链、结构化输出 | `qa_report.json` |
| Document Writer Agent | 写成固定模板文档 | 文档生成与一致性校验、JSON Schema 输出 | `character_bible.md` |

## 7. 10 个 skill 的推荐定义方式

这些 skill 不要写成“玄学 prompt 名词”，而要写成稳定的、可测试的能力模块。

### 7.1 建议把 skill 分成三类

第一类，确定性技能：

- `人名识别`
- `章节/场景出场统计`
- `证据回链`
- `JSON Schema 结构化输出`

第二类，NLP + 规则混合技能：

- `别名归并`
- `指代消解`
- `台词归属`
- `人际关系抽取`

第三类，LLM 主导技能：

- `人物属性抽取`
- `文档生成与一致性校验`

### 7.2 每个 skill 都要有固定输入输出

每个 skill 最少要定义：

- 输入 schema
- 输出 schema
- 失败状态
- 置信度
- 证据列表
- 版本号

推荐字段：

```json
{
  "skill_name": "alias_merge",
  "skill_version": "v1",
  "input": {},
  "output": {},
  "confidence": 0.0,
  "evidence": [],
  "warnings": [],
  "needs_review": false
}
```

### 7.3 不要让 skill 直接输出“最终真相”

例如 `别名归并` 不应该直接说“这两个一定是同一个人”。

更稳的输出方式是：

- 候选并集
- 置信度
- 支持证据
- 反证
- 是否需要人工确认

这样 `Consistency QA Agent` 才有空间兜底。

## 8. 中英文两套推荐落地方案

### 8.1 中文小说

推荐：

- `HanLP + LLM 抽取 + LangGraph`

原因：

- `HanLP` 当前仍然是中文生产环境里最稳的一类通用 NLP 工具包
- 官方仓库明确覆盖命名实体识别、依存、SRL、指代消解等能力
- 中文小说里真正难的是：
  - 别名丰富
  - 称谓和关系称呼复杂
  - 省略主语严重
  - 台词归属常常不显式

所以中文路线必须承认现实：

- `HanLP` 负责尽量多的基础结构化
- `LLM + 规则` 负责 alias merge、关系理解、人物属性总结
- `QA Agent` 负责压制 hallucination

### 8.2 英文小说

推荐：

- `BookNLP + LangGraph`

原因：

- `BookNLP` 本身就是为书籍和长文档设计的 pipeline
- 官方仓库直接支持：
  - entity recognition
  - character name clustering
  - coreference resolution
  - quotation speaker identification
  - event tagging
- 它还能直接产出 `.entities`、`.quotes`、`.book` 等结构化文件

因此英文路线比中文更省力：

- 可以把 `BookNLP` 作为前处理主干
- LLM 只负责高阶总结、导演对齐和 QA

## 9. 为什么不建议把“全流程人物系统”直接做成纯多 Agent 对话

因为这种系统的失败方式很明确：

- 人物发现阶段漏了一个角色
- alias merge 阶段把两个人错并
- 对白归属阶段台词给错人
- profile extraction 阶段把文学解读写成事实
- document writer 把前面错误包装得更漂亮

如果没有中间结构和证据层，后面所有 agent 都会把错误越滚越大。

所以稳妥路线是：

- 先做结构
- 再做解释
- 最后做文档

而不是一开始就让 7 个 agent 在聊天里互相传摘要。

## 10. `OpenClaw` 在人物系统里的落地点

### 10.1 建议接入的位置

适合让 `OpenClaw` 做的事：

- 登录内部网页知识库，抓导演分析稿
- 打开 CMS / Notion / 飞书文档，把分析稿导出到本地
- 把最终人物文档回填到协作平台
- 给人工审核台做网页截图、状态采集、下载
- 把一组固定网页动作封成项目私有 skill

### 10.2 不建议接入的位置

不建议让 `OpenClaw` 直接负责：

- 章节切分
- 人名识别
- 指代消解
- alias merge 主逻辑
- 人物画像主逻辑
- QA 主逻辑

这些工作本质上更适合：

- 本地 Python 能力
- 可测试的 schema
- 可持久化的中间结果

### 10.3 如果接 `OpenClaw`，推荐怎么接

```text
LangGraph Orchestrator
  -> local NLP / LLM extractors
  -> local QA / doc writer
  -> OpenClaw skill node
       -> director-note-fetch
       -> review-console-sync
       -> report-publish
```

同时保持边界：

- `OpenClaw` 只处理网页型动作
- 登录坚持人工完成
- 第三方 skill 视为不可信代码
- 项目优先使用 workspace 私有 skill

## 11. 一版可落地的数据模型建议

### 11.1 核心实体

`character_registry`

```json
{
  "character_id": "char_0001",
  "canonical_name": "张三",
  "aliases": ["三哥", "老张"],
  "titles": ["张队", "张老板"],
  "first_appearance": {
    "chapter_id": "ch_001",
    "scene_id": "sc_001",
    "span_id": "span_000123"
  },
  "confidence": 0.93
}
```

`evidence_link`

```json
{
  "evidence_id": "ev_0001",
  "book_id": "book_xxx",
  "chapter_id": "ch_001",
  "scene_id": "sc_001",
  "span_start": 1024,
  "span_end": 1088,
  "quote": "张三抬头看了她一眼。",
  "source_type": "novel_text"
}
```

`profile_fact`

```json
{
  "character_id": "char_0001",
  "fact_type": "personality",
  "fact_value": "谨慎、寡言",
  "evidence_ids": ["ev_0102", "ev_0131"],
  "confidence": 0.74,
  "needs_review": true
}
```

### 11.2 为什么先用关系表而不是图数据库

第一版不建议直接上图数据库。

原因：

- 图关系并不是这个系统第一阶段的瓶颈
- 第一阶段瓶颈是“抽对”和“可回链”
- `Postgres + JSONB + join tables` 足够支撑绝大多数人物关系需求

等人物实体稳定后，再考虑把关系边同步到图数据库。

## 12. 分阶段实施建议

### 阶段 0：单书离线跑通

目标：

- 不接任何网页
- 单本小说离线跑通全链路
- 输出人物总表、单人人设卡、关系表、代表台词、证据片段

技术建议：

- `LangGraph`
- `SQLite`
- 中文走 `HanLP`
- 英文走 `BookNLP`

### 阶段 1：加入人工审核断点

目标：

- alias merge 人工确认
- QA 冲突人工确认
- 导演稿对齐人工确认

技术建议：

- `LangGraph interrupt`
- 简单审核页面或 CLI 审核命令

### 阶段 2：接入导演分析稿与外部系统

目标：

- 读取导演分析稿
- 形成“客观文本证据 + 导演诠释”双层描述
- 回写到团队协作平台

技术建议：

- 必要时用 `OpenClaw`
- `OpenClaw` 只做网页采集和回填

### 阶段 3：做服务化和批处理

目标：

- 多本书批量处理
- 多版本角色档案
- 可追踪审校记录

技术建议：

- `Postgres`
- 队列系统可后加
- `Flowise` 或自研前端可作为运营台，不要反客为主

## 13. 最终推荐排序

### 13.1 作为人物系统主干

1. `LangGraph`
2. `Haystack`
3. `LlamaIndex`

### 13.2 作为多角色原型工具

1. `AutoGen`
2. `CrewAI`
3. `Flowise`

### 13.3 作为网页执行和 skill 平台

1. `OpenClaw`

### 13.4 当前最现实的两条落地线

中文：

- `HanLP + LLM 抽取 + LangGraph`

英文：

- `BookNLP + LangGraph`

## 14. 我对你当前项目的直接建议

如果你现在就开做，我建议不要再花时间找“某个万能多 agent 框架”。

更实际的路线是：

1. 先定 `LangGraph` 为主编排
2. 把 7 个 Agent 的输入输出 schema 定死
3. 中文先用 `HanLP + 规则 + LLM`
4. 英文先用 `BookNLP`
5. 先做离线单书跑通
6. `OpenClaw` 只在“导演稿网页采集 / 审核回填 / 发布”阶段接入

这是当前最成熟、最稳重、工程风险最低的落地方式。
