# VideoOnlyOnce 小说分类扩展与知识库 Agent 设计

## 1. 目的

原先的阶段 0 任务已经有基础方向，但分类维度还偏少，容易出现两个问题：

- 只能识别“大题材”，不能识别网文里真正决定画风和场景的细分流派
- 读得越多，知识却没有沉淀成可复用的项目资产

因此建议把这部分拆成两层：

- 一层是“扩展后的小说分析任务类别”
- 一层是“持续读取章节并维护知识库的专用 Agent”

## 2. 基于主流小说平台补充出来的分类维度

我在 2026-03-26 查阅了主流官方平台页面，能确认当前网文平台的分类已经远不止“古代修仙、现代修仙、诡异、都市异能”这几个大类。

### 2.1 平台能提供什么价值

- 起点中文网：
  - 主干分类非常稳定，长期覆盖玄幻、仙侠、都市、游戏、轻小说等大类
  - 玄幻、仙侠、都市下有明显细分风向，例如传统玄幻、修真文明、都市修仙、灵气复苏
- 17K：
  - 创作中心的官方入门指南对题材拆分最适合抽成规则
  - 明确提到仙侠可拆为洪荒封神、古典仙侠、现代修真
  - 玄幻常见流派包括废柴流、强者归来流、系统流
  - 都市常见流派包括兵王、医生、奶爸、都市修仙、赘婿、鉴宝、灵气复苏、战神回归
- 番茄小说：
  - 最近的官方赛道设计更接近“平台真实流量风向”
  - 新玄幻里明确区分东方仙侠、传统玄幻、玄幻言情
  - 还把悬疑灵异、女频悬疑、现言脑洞、古风世情、宫斗宅斗等拆得很细
- 纵横中文网：
  - 官方活动页能看到男频长期稳定的重点赛道，如玄幻、都市、历史、仙侠、奇幻、武侠、悬疑、奇闻异录、二次元、现实题材
- 晋江文学城：
  - 官方读者帮助页强调除了“类别”还有“属性”，这对我们做标签系统非常重要
- 七猫中文网：
  - 女频侧分类常见“玄幻仙侠、都市奇幻、宫闱宅斗、古代情缘、权谋天下、年代重生”等，适合作为女频视觉标签补充

## 5. 为什么需要单独做一个知识库 Agent

如果每次都让分镜 Agent 临时读前 30 章，再临时总结，它会有 4 个问题：

- 读得越多，上下文越贵
- 同一个角色会被反复重述，但不稳定
- 之前提炼过的场景锚点不会自动复用
- 后来新章节推翻旧设定时，没有版本管理

所以建议单独做一个常驻 Agent，职责不是“出分镜”，而是“持续做小说知识沉淀”。

## 6. Agent 角色定义

建议命名：

- `novel_knowledge_agent`
- 或 `story_bible_agent`

它的职责只有 5 件事：

- 读取新章节
- 提取新增设定
- 和旧知识库比对
- 更新知识库
- 向下游导出“给分镜用的精简上下文”

它不要直接负责出图、出视频、改 prompt。

## 7. Agent 的输入、输出与状态

## 7.1 输入

- 小说章节原文
- 已有知识库
- 风格预设库
- 术语表
- 人工修订记录

## 7.2 输出

- `story_bible.md`
- `character_bible.md`
- `scene_bible.md`
- `visual_motifs.md`
- `audio_motifs.md`
- `continuity_rules.json`
- `chapter_extractions/ch001.json`
- `chapter_extractions/ch002.json`
- `change_log.jsonl`
- `export/context-pack-ep01.json`

## 7.3 状态层级

- `candidate`
  - 本章首次出现，暂未确认
- `confirmed`
  - 连续多章出现，或被主叙事明确确认
- `deprecated`
  - 后续章节推翻或废弃
- `ambiguous`
  - 有歧义，等待人工确认

## 9. Agent 的工作流

## 9.1 首次建库

1. 读取前 30 章。
2. 先做粗分类。
3. 再做角色、场景、力量体系、视觉母题抽取。
4. 生成初版 `story_bible` 系列文档。
5. 生成人工 review 清单。
6. 人工确认后标记为 `v1`。

## 9.2 增量更新

1. 读取新章节。
2. 只提取新增或冲突信息。
3. 和已有实体表做 canonical merge。
4. 写入 `chapter_extractions/chXXX.json`。
5. 记录 `change_log.jsonl`。
6. 若某元素多次出现或被正文明确坐实，则从 `candidate` 提升为 `confirmed`。
7. 重新导出下游 context pack。

## 9.3 给下游提供什么

不要把整本文档直接塞给分镜 Agent，而是导出一个精简包：

- 当前集涉及的角色
- 当前集涉及的场景
- 当前集涉及的视觉母题
- 当前集禁用项
- 当前集默认风格
- 当前集对白口吻和声线建议

## 10. Agent 的核心抽取策略

建议不是只做一次大总结，而是分 4 类抽取器。

### 10.1 实体抽取器

负责抽：

- 人物
- 势力
- 地点
- 法器
- 异兽
- 组织

### 10.2 世界观抽取器

负责抽：

- 修炼体系
- 历史背景
- 阵营秩序
- 世界地图
- 禁忌规则

### 10.3 视听母题抽取器

负责抽：

- 颜色
- 光线
- 材质
- 声音
- 天气
- 图腾
- 高频动作

### 10.4 连续性审计器

负责查：

- 角色外观是否漂移
- 地点描写是否冲突
- 法器设定是否变更
- 时间线是否矛盾
- 口头禅和声线是否漂移

## 12. 这个 Agent 应该怎么接到现有仓库

最稳的做法：

- 它挂在 `fenjing_program` 下
- 不直接放进 `zaomeng`

原因：

- `fenjing_program` 天然已经是前置理解和结构化输出层
- `zaomeng` 更适合执行浏览器生成，不适合做长程知识沉淀

### 建议新增命令

- `intake-novel <book_id> --chapters 1:30`
- `update-kb <book_id> <chapter_file>`
- `review-kb <book_id>`
- `export-context <book_id> <episode_id>`
- `diff-kb <book_id> <chapter_file>`

### 建议新增模块

- `src/feicai_seedance/novel_intake.py`
- `src/feicai_seedance/knowledge_base.py`
- `src/feicai_seedance/entity_extractors.py`
- `src/feicai_seedance/context_exporter.py`
- `src/feicai_seedance/continuity_auditor.py`

## 13. Agent 的落地规则

## 13.1 先规则，后模型

第一版不要全靠 LLM 幻觉推断。

先做：

- 明确字段 schema
- 明确状态机
- 明确提取模板
- 明确 merge 规则

再让 LLM 只负责填充候选值。

## 13.2 先追加，后覆盖

不要让 Agent 直接重写整本知识库正文。

建议：

- 每章先写 `chapter_extractions/chXXX.json`
- 再生成 `change_log`
- 最后由 merge 步骤更新 canonical knowledge base

## 13.3 先 candidate，后 confirmed

单章出现一次的设定，不要立刻当铁律。

推荐提升规则：

- 连续 2 到 3 章重复出现
- 或被作者明确解释
- 或被主情节强绑定

## 13.4 冲突必须显式记录

比如：

- 第 5 章说角色穿青袍
- 第 16 章说角色常穿黑袍

这种不能 silently overwrite，必须：

- 记录冲突
- 打上时间戳
- 等人工 review 或更高置信度章节解决

## 14. 给分镜 Agent 的最佳协作方式

建议采用双 Agent：

- `novel_knowledge_agent`
  - 长期维护小说世界知识库
- `storyboard_agent`
  - 每次只消费当前 episode 的精简 context pack

这样可以避免两种问题：

- 分镜 Agent 上下文过大
- 知识库 Agent 被临时镜头任务污染

## 15. 推荐的最小落地版本

V1 不要追求智能到“自动懂一切”，只做下面这些就够：

1. 支持前 30 章批量 ingest。
2. 支持角色、场景、法器、势力四类实体抽取。
3. 支持世界观类型和细分流派标签。
4. 支持视觉母题和禁用项抽取。
5. 支持导出 `context-pack-ep01.json` 给分镜阶段使用。

## 16. 结论

你这个判断是对的：

- “小说分类体系要扩”
- “小说知识提取最好单独做 Agent”

真正可落地的做法不是让分镜阶段顺手总结一点，而是：

- 从平台真实分类里抽出更细的流派标签
- 让一个专门的 `novel_knowledge_agent` 持续读章节、提取设定、维护知识库
- 再把精简后的 context pack 交给分镜、出图、视频、TTS 阶段消费

这样后面章节越多，系统不是越乱，而是知识库越稳。
