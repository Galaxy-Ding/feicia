# 需求报告

## 1. 背景

当前仓库已经存在两个子系统：

- `fenjing_program`
  - 负责文本理解、导演分析、角色/场景设计、Prompt 结构化产物、审核 gate
- `zaomeng`
  - 负责网页自动化、图片下载归档、浏览器执行和阶段状态

问题不在于“没有代码”，而在于：

- 两套系统边界清楚，但还没有统一中控工作区
- 文档和实施节奏是分开的，后续扩大后容易失控
- 阶段总结、问题日志、测试计划、验收计划没有在根目录形成统一基线
- 后续要做自动化升级时，容易出现模块重复建设和目录分裂

## 2. 本期目标

第一阶段目标不是做完整产品，而是建立后续改造的统一起点。

必须完成：

- 建立根目录统一文档包
- 建立统一 phase1 代码目录
- 将 `fenjing_program` 和 `zaomeng` 纳入统一集成入口
- 定义统一任务契约、路径契约和安全边界
- 建立统一测试基线
- 建立阶段总结与问题闭环机制

## 3. 业务需求

### BR-01 统一工程入口

- 后续改造必须从统一工作区进入
- 不再要求使用者自己在两套子工程之间手动判断执行顺序

### BR-02 分阶段可实施

- 每一阶段都必须能单独交付、单独验证、单独总结
- 新阶段开始前必须回顾上一阶段总结

### BR-03 问题闭环

- 所有阶段问题必须统一进入问题日志
- 同时输出阶段问题点，便于短期执行

### BR-04 可持续扩展

- 未来可以继续接知识库 Agent、角色锚定、视频链路、TTS、渲染链路
- 但不应推翻第一阶段目录和契约

## 4. 工程需求

### FR-01 统一代码目录

- 在根目录建立单独的 phase1 代码文件夹
- 该目录负责统一调度 `fenjing_program` 和 `zaomeng`
- 该目录必须具备自己的 `README`、`pyproject`、`src/` 和 `tests/`

### FR-02 统一路径契约

- 能清晰定位：
  - 项目根目录
  - `fenjing_program`
  - `zaomeng`
  - 集成输出目录
  - 集成日志目录
  - 集成配置目录

### FR-03 统一任务契约

- 至少定义 episode 级任务元数据
- 至少定义阶段 gate 决策结果
- 至少定义下游命令桥接参数

### FR-04 统一 CLI 集成骨架

- 能输出统一状态
- 能生成或展示子系统执行命令
- 能创建统一工作区目录

### FR-05 测试基线

- 每个新增单元函数至少有 3 组随机测试样例
- 必须补齐：
  - 单元测试
  - 功能测试
  - 集成测试
  - 安全测试

### FR-06 阶段文档闭环

- 每阶段必须产出：
  - 阶段总结
  - 阶段问题点
  - 更新后的问题日志

## 5. 非功能需求

- 低耦合：集成层不直接侵入两个子系统内部实现
- 可审计：所有阶段状态和问题有据可查
- 可维护：目录和命名规则稳定
- 可安全：路径处理必须防止越界访问
- 可演进：后续可从 CLI 过渡到状态图编排

## 6. 本期成功标准

- 根目录形成统一文档包
- 形成单独的 phase1 代码目录
- phase1 代码目录可识别两个子系统并输出统一状态
- 测试通过
- 阶段总结、阶段问题点、问题日志已形成闭环
