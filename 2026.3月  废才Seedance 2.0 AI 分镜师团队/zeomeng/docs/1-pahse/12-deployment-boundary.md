# 部署边界说明

## 1. 文档目的

本文件用于单独说明一期项目中的部署边界，避免团队在实施时混淆“技术选型”和“当前机器部署要求”。

一期已经明确采用以下技术选型：

- `OpenClaw`
- `browser skill`
- 本地文件处理脚本

但本期同时明确：

- 当前这台电脑不部署 OpenClaw

## 2. 当前部署结论

### 2.1 结论

一期项目在当前阶段不要求在本机安装、部署或常驻运行 OpenClaw。

### 2.2 原因

- 当前存在安全顾虑，不适合在这台电脑直接部署 OpenClaw
- 一期当前核心目标是冻结需求边界、明确调用方式和准备配套本地目录与脚本
- 提前部署 OpenClaw 会引入额外的安装、权限、网络和运行时风险

## 3. 技术选型与当前落地的区别

### 3.1 技术选型

技术选型回答的是“未来正式方案用什么框架”。

一期已确定：

- 浏览器自动化框架使用 `OpenClaw`
- 浏览器操作能力通过 `browser skill` 承担
- 下载后重命名、归档、日志由本地文件处理脚本承担

### 3.2 当前机器落地

当前机器回答的是“现在这台机器能先做什么”。

当前已经完成：

- 不部署 OpenClaw 的前提下建立本地代码骨架
- 落地配置模板、选择器模板、提示词模板
- 落地任务编排、状态追踪、日志、归档、映射记录
- 通过模拟浏览器跑通一期闭环

## 4. 后续如何对接 OpenClaw

当后续安全条件允许、或切换到可部署的机器环境时，可以按以下方式对接 OpenClaw。

### 4.1 对接目标

- OpenClaw 负责调度 workflow
- browser skill 负责即梦网页的打开、定位、输入、点击、等待和下载
- 本地文件处理脚本负责下载监听、重命名、映射记录和日志

### 4.2 当前已预留的对接边界

浏览器侧接口：

- `validate_login()`
- `open_generation_page()`
- `submit_prompt()`
- `wait_for_generation()`
- `download_images()`

本地脚本侧接口：

- 文件稳定性判断
- 文件归档与重命名
- 映射记录输出
- 运行日志与错误日志
- 任务状态持久化

## 5. 当前机器已完成的本地准备

### 5.1 目录与模板

已建立并固定以下目录：

- `workflow/configs/`
- `workflow/selectors/`
- `workflow/prompts/`
- `downloads/staging/`
- `downloads/images/`
- `logs/runs/`
- `logs/errors/`
- `state/tasks/`
- `state/browser/`

### 5.2 已完成配置模板

已落地：

- 图片页选择器模板
- 下载目录配置
- 命名规则配置
- 超时与轮询配置
- 提示词任务文件模板

### 5.3 已完成本地脚本

本机已经实现：

- 任务编排脚本
- 提示词加载脚本
- 文件稳定性判断脚本
- 图片重命名脚本
- 原始文件名与新文件名映射记录脚本
- 执行日志与错误日志写入脚本
- 任务状态持久化脚本
- 模拟浏览器适配器

### 5.4 已完成验证

当前已完成：

- 本地模拟链路运行
- 下载样例生成
- 映射记录生成
- 自动化测试通过
- 阶段总结与问题日志更新

## 6. 当前机器不做的事项

- 不安装 OpenClaw
- 不运行 OpenClaw 服务
- 不做真实 browser skill 联调
- 不在本机接管即梦网页实际浏览器会话
- 不处理 OpenClaw 的安全加固与运行隔离

## 7. 风险与建议

### 7.1 当前风险

- 文档和本地脚本先行，可能与未来真实页面细节存在偏差
- 没有实际 browser skill 联调前，选择器有效性无法最终确认
- 当前还缺少真实截图证据

### 7.2 建议

- 当前阶段继续把“目录、配置、命名、日志、接口边界”冻结
- 后续切换到安全可控机器时，再进行 OpenClaw 实机对接
- 对接时优先跑单条提示词样例，不要一开始就批量执行

## 8. 最终结论

一期项目的目标框架仍然是：

- `OpenClaw + browser skill + 本地文件处理脚本`

但在当前阶段：

- 本机不部署 OpenClaw
- 本机已完成方案、配置、目录、脚本、测试和文档准备
- 真正的 OpenClaw 接入和浏览器联调放到后续安全条件允许的环境中执行
