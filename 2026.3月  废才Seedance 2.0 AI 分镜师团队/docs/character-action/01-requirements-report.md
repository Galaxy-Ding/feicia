# 需求报告

## 1. 背景

当前团队需要把中文和英文小说的人物角色识别、归并、关系抽取和证据回链能力，沉淀成一套可重复执行、可人工审核、可持续扩展的工程系统。

直接用单次 LLM 调用处理整本小说的问题很明显：

- 长文本上下文不稳定
- 角色别名和指代容易漂移
- 无法保留中间状态
- 无法可靠回溯证据
- 缺少人工修正后的持续生效机制

因此系统必须采用“确定性预处理 + 有状态编排 + 结构化抽取 + 审核闭环”的路线。

## 2. 本期目标

本期目标是建立一套双语统一的人物角色系统实施基线。

必须完成：

- 明确中文小说处理链路
- 明确英文小说处理链路
- 明确 LangGraph 主编排设计
- 明确 HanLP / BookNLP 的职责边界
- 明确规则抽取器与 LLM 抽取器的分工
- 明确 JSON Schema 输出契约
- 明确 SQLite 与 Postgres 的存储策略
- 明确人工审核、重跑、回写与验收标准

## 3. 业务需求

### BR-01 双语统一

- 中文小说与英文小说必须共用统一上层业务模型
- 语言差异只允许体现在底层预处理器与局部规则中

### BR-02 人物全生命周期管理

- 支持人物发现
- 支持别名归并
- 支持人物属性抽取
- 支持人物关系抽取
- 支持证据片段回链
- 支持人工修正后再次入图

### BR-03 面向后续分镜系统

- 输出的人物卡必须适合作为后续分镜、镜头规划和视觉资产锚定的输入
- 数据结构必须稳定，不能每本书都靠 prompt 临时决定字段

### BR-04 可中断可恢复

- 整本小说处理必须支持章节级、批次级中断恢复
- 人工审核后必须支持从指定节点继续执行

### BR-05 可审计

- 所有角色结论必须能定位到原文 chunk、句子或 span
- QA 失败项必须保留原因和处理记录

## 4. 功能需求

### FR-01 书籍接入

- 支持录入书籍基础元数据
- 支持分章节导入
- 支持语言识别或显式指定语言

### FR-02 文本预处理

- 中文链路使用 `HanLP`
- 英文链路使用 `BookNLP`
- 统一输出章节、段落、句子、token、候选实体、指代线索

### FR-03 抽取与归并

- 基于规则抽取器做第一轮高精度候选生成
- 基于 LLM 抽取器做补全与复杂推理
- 基于归并器生成 canonical character 与 alias 集合

### FR-04 关系与属性

- 抽取基础属性：姓名、性别倾向、身份、阵营、首次出场、典型描写
- 抽取关系：亲属、师徒、同伴、敌对、上下级、情感关系
- 输出关系置信度与证据

### FR-05 审核与质检

- 提供人工审核任务队列
- 支持低置信度项进入审核
- 支持 QA 规则自动发现冲突

### FR-06 持久化与查询

- 支持 SQLite 本地开发
- 支持 Postgres 生产部署
- 支持按书籍、章节、人物、别名、关系、证据进行查询

## 5. 非功能需求

- 可扩展：后续可接入更多语言与更多领域规则
- 可维护：Schema、数据库、图状态均需版本化
- 可测：每个处理层必须有独立测试入口
- 可恢复：任何长流程节点必须支持 checkpoint
- 可追责：所有人工变更必须有审计记录
- 可迁移：SQLite 与 Postgres 间契约保持一致

## 6. 本期成功标准

- 已形成完整方案级文档包
- 已明确双语统一数据模型
- 已明确实施模块、任务拆分、测试计划和验收门槛
- 已定义实施阶段的主要风险与问题日志基线
